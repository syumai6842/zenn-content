---
title: "成果物の品質をループで保証する"
---

## reporterだけでは足りなかった

初期は、CEOがreporterを直接起動してHTMLレポートを生成していました。reporterは執筆と自己レビューを一体で行います。

問題は、reporter自身が自分のデザインを客観的に評価できないことでした。毎回似たような色、似たようなレイアウト、似たようなフォントの成果物が出てきます。「ありきたり」を検出する仕組みがありませんでした。

## design-loopの導入

解決策として、reporterとCEOの間にdesign-loopを挟みました。

```
CEO
  └── design-loop（デザインループオーナー）
        ├── reporter（コンテンツHTML生成）
        └── designer（ビジュアルデザイン適用）
```

design-loopの仕事は「全デザイン要素を根拠ベースで評価し、改善ループを回す」ことです。

### 実行フロー

```
1. reporterにコンテンツHTMLを生成させる
2. designerにビジュアルデザインを適用させる
3. デザイン監査（全要素を評価）
4. 不合格要素を修正指示 → 2に戻る（最大3ラウンド）
5. 全要素合格 → 最終HTML出力
```

### 「根拠ベース」とは

デザインの選択に対して「なぜこの色か」「なぜこのフォントか」「なぜこのレイアウトか」を問います。「きれいだから」は根拠ではありません。「データ密度を重視するレポートでは、monospace fontがスキャンしやすい（参照: Bloomberg端末のデザイン原則）」が根拠です。

## デザインシステム

成果物のスタイルはデザインシステムで管理しています。

```
templates/
  report-base.css      # 共通基盤（タイポグラフィ・グリッド・コンポーネント）
  themes/
    analysis.css       # デフォルト: ターミナル密度 + オレンジ発光アクセント
    sophisticated.css   # ゴールド系・高級感
```

report-base.cssには以下のコンポーネントが含まれます。

- metric-change（増減表示）
- progress-bar
- comparison-grid
- timeline
- tag/chip
- inline-bar

Chart.js（CDN）を使ったデータビジュアライゼーションも標準で含んでいます。折れ線・棒・ドーナツ・レーダー・積み上げ棒をデータに応じて使い分けます。

デフォルトのanalysisテーマは黒背景(#050508)にオレンジアクセント(#f07020)、JetBrains Monoフォントで、Bloomberg端末的なデータ密度を意図しています。

## designerエージェント

designerはビジュアルデザイン専門のworkerです。

```yaml
model: sonnet
tools: Read, Write, Edit, WebSearch, WebFetch
```

designerの仕事は、report-base.cssの上でCSS変数オーバーライドとカスタムスタイルを追加することです。ゼロからデザインするのではなく、デザインシステムの上で用途に応じた調整を行います。

「テンプレ感のないデザイン」が指示として含まれています。デフォルト設定をそのまま使うのではなく、用途・対象読者・文脈に応じた意図的な選択を行うことを求めます。

## reporterの責務

reporterはコンテンツHTMLの生成に特化しています。

```yaml
model: opus
tools: Read, Write, Bash
```

reporterにopusを使うのは、大量のリサーチ結果から情報の取捨選択（重複の統合）と構造化を行う必要があるためです。ただし、取捨選択で事実を削ることは禁止しています。削っていいのは重複だけです。

## 出力の原則

成果物生成全体を通じて、以下の原則を適用しています。

- 情報を落とさない（網羅性最優先）
- 各事実に出典と数値の前提条件を添える
- 評価語・示唆は足さない
- 比喩・キャッチコピー化しない
- 根拠のない評価語を使わない
- 切りのいい数値の演出をしない

これらはすべてpreferences.mdに記録されており、セッション開始時に自動で読み込まれます。
