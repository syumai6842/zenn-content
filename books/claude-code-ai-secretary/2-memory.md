## 問題: セッションが切れると記憶が消える

Claude Codeのセッションは独立しています。昨日の会話の内容を今日のセッションは知りません。毎回「このプロジェクトの背景は...」と説明し直すのは、秘書としては致命的です。

解決策は「メモリファイル」です。Claude Codeは起動時にCLAUDE.mdと、そこから参照されるファイルを読み込みます。ここに記憶を書き込めば、セッションをまたいで情報が持続します。

しかし単にファイルを置くだけでは「手動で更新しなければならないメモ帳」にしかなりません。自律的に育つ仕組みが必要です。

## メモリの構造

メモリは役割別に分離しています。

```
~/.claude/memory/
  glossary.md      # 用語集（略語・コードネーム）
  people.md        # 人物録（名前・役割・関係）
  preferences.md   # 好み・習慣・スタイル
  decisions.md     # 意思決定ログ（ADRの個人版）
  lessons.md       # 失敗ログ・学びログ
  handoff.md       # セッション間の引き継ぎ
  projects.md      # プロジェクトインデックス
```

分離する理由は2つあります。

1つ目は**参照コストの制御**。CLAUDE.mdに全部書くとコンテキストウィンドウを圧迫します。CLAUDE.mdには「必読メモリ」（glossary、people、projects、preferences）だけを常時参照として指定し、decisions.mdやlessons.mdは「オンデマンド参照」として必要なときだけ読むようにしています。

2つ目は**更新の粒度**。「新しい人物を追加する」操作は people.md だけに閉じます。1ファイルに全部入っていると、追記のたびに他の情報を巻き込むリスクがあります。

## 自律学習プロトコル

ここが核心です。メモリの更新をユーザーが手動で行うのではなく、Claude Codeが会話の中で自動的に検出して書き込みます。

CLAUDE.mdに以下のルールを書いています。

```markdown
会話の流れの中で以下のいずれかを検出したら、
ユーザーに確認せず該当ファイルを Edit / Write ツールで直接更新する。

| 検出した情報 | 更新先 | 操作 |
|-------------|--------|------|
| 新しい社内用語/略語 | glossary.md | 追記 |
| 新しい人物 | people.md | 追記 |
| プロジェクトの状態変化 | Obsidian | 更新 |
| 表明された好み | preferences.md | 追記または上書き |
| 重要な意思決定 | decisions.md | 追記 |
| 学び・気づき | lessons.md | 追記 |
```

ポイントは「**ユーザーに確認せず**」の部分です。「これをメモリに追加していいですか？」と毎回聞かれたら、手動更新と変わりません。`settings.json` でmemoryディレクトリへの書き込みを事前承認し、確認ダイアログが出ないようにしています。

## 自動更新のガードレール

確認なしで書き込む以上、暴走を防ぐルールが必要です。

```markdown
1. 重複は避ける: 追記前に既存内容を読み、似た記述があれば統合する
2. 確信ベース: 曖昧な情報は書かない。明確に表明されたもののみ
3. 報告は最小限: 重要な追加のみ1行報告。些末な追記は黙って実行
4. 1エントリずつ追記: まとめて巨大な更新はしない
5. 削除は提案のみ: 自動で削除はしない
```

特に重要なのは5番です。追記は自動でやるが、削除は人間に確認する。これにより「間違えて書いたものを消せなくなる」事態を防ぎます。

## decisions.md — ADRの個人版

意思決定ログは特に重要なメモリです。形式はADR（Architecture Decision Record）を個人用に簡略化したものです。

```markdown
## YYYY-MM-DD: 方針転換の例
- 状況: あるプロジェクトの調査方針を検討していた
- 決定: デスクリサーチ中心からフィールド調査併用に切り替え
- 理由: 一次情報の質がデスクリサーチと段違いだった
```

なぜこれが必要かというと、3ヶ月前の自分がなぜその判断をしたかを、AI秘書が正確に説明できる必要があるからです。「なんでこの方針なんだっけ？」と聞いたとき、decisionsを引いて根拠ごと答えられます。

覆された決定は四半期ごとにアーカイブファイルに移動し、現行の決定だけがアクティブなメモリに残るようにしています。

## lessons.md — 失敗ログ

もう1つ重要なのが失敗ログです。これは第5章で詳しく扱いますが、構造だけ先に紹介します。

```markdown
## YYYY-MM-DD: 「日本に不在」判定の構造的誤り
- 何が起きた: 「日本に特定業種向けSaaSは確認できない」と
  記載したが、深掘りで複数存在が判明
- 何を学んだ: 問いの立て方自体が確証バイアスを生む
- 次にどうする: 「日本のユーザーは今どう解決しているか」で問いを立てる
```

このログが蓄積されることで、同じ種類のミスを2度目は検出できるようになります。実際に「不在結論の検証」は、2日連続で同じ誤りが出たことをきっかけにガードレール化されました。

## handoff.md — セッション間の引き継ぎ

handoff.mdは「昨日の自分から今日の自分へ」のメモです。セッション開始時のhookで自動的にコンテキストに注入されます。

内容は以下の構成です。

- 直近の状況（現在の方針・進行中のプロジェクト・最新の決定）
- 未完了タスク
- 朝一で確認すべきこと
- 注意・気を付けること

このファイルは追記ではなく**上書き**で運用します。常に最新の状態だけが残り、過去のhandoffは不要です（過去の記録はdecisions.mdとlessons.mdに分散保持されている）。

## preferences.md — 好みの蓄積

ユーザーの好みは明示的に伝えなくても会話の中で現れます。

例えば「比喩を使わないで」と一度言えば、preferences.mdに「比喩・メタファー禁止」が追記されます。「根拠のない評価語（効果的・有効・重要）は使わないで」と言えば、それも追記されます。

2ヶ月運用した結果、preferences.mdには以下のようなルールが蓄積されました。

- 文章スタイル（絵文字なし、前置きなし、キャッチコピー化しない）
- データ表示の原則（分類軸を先に定義しMECEを確認してから分ける）
- アウトプットの原則（情報を落とさない、網羅性最優先）
- CEO/subagent運用の役割分担（分析・解釈はユーザーが行う）

これらは1つ1つが実際の会話で「それは違う」と訂正された結果です。手動でまとめて書くよりも、実運用の中で少しずつ正確になっていく方が結果的に精度が高くなります。

## 300行制限とアーカイブ

メモリファイルが肥大化するとコンテキストを圧迫します。各ファイルに300行程度の上限を設け、超えたらarchivistエージェントに整理・圧縮を委譲する運用にしています。decisions.mdは四半期ごとに覆された決定をアーカイブファイルに移動します。

要点: メモリは「書く」より「捨てる」設計の方が難しいです。蓄積は自動でできますが、何を残し何を捨てるかの判断には人間のレビューが必要です。
