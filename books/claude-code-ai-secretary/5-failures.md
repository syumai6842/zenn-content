## なぜ失敗ログを残すのか

AI秘書の品質は「何ができるか」ではなく「何を間違えないか」で決まります。毎回正しい答えを出すのは無理ですが、同じ間違いを2回しないことはシステムで保証できます。

lessons.mdに蓄積された失敗ログの中から、運用設計に影響を与えた主要なものを紹介します。

## 分類1: 確証バイアス・問いの立て方

### 「日本に不在」判定の構造的誤り（2件連続）

最も影響が大きかった失敗です。

**1件目（2026-06-29）**: 海外トレンドの並列スキャンで「日本にペットグルーミング特化のSaaSは確認できない」と記載。深掘りで複数存在が判明。

**2件目（2026-06-30）**: 音楽D2Cの仮説で「日本にEVEN型（音楽直販+購入者メール取得）は不在」と結論。BASE/STORES/Bandcamp/Gumroadで要件を満たすサービスが複数存在していた。

**根本原因**: 「日本にこのプロダクトがあるか/ないか」という問いの立て方自体が確証バイアスを生む。「ない」が出ると事業仮説が前に進むため、検証動機が構造的に弱くなる。

**対策**: リサーチの問いを「○○は存在するか」から「**日本のユーザーは今この問題をどう解決しているか**」に変更。research-loopに`absent-unverified`状態を追加。

### ベンダー側マーケ記事を需要側painの根拠に使った

ビデオ接客の検討で「小売・ブランド側にpainがある」と書きましたが、根拠はLiveCall/NTT東日本等のベンダー側マーケ記事でした。需要側の独立調査は見つかりませんでした。

ベンダーは自分の市場を大きく見せる動機があります。「成長率34%」は導入率の低さ（ベース小さい）と表裏であり、絶対値での需要規模を示しません。

**対策**: 「painがある」と書くときは、ベンダー外の独立ソースを1本以上引く。引けない場合は「ベンダー側報道のみ・需要側未検証」と明記する。

## 分類2: 独断・過剰介入

### 3Dモデルの上下を独断で補正した

ポートフォリオサイトのthree.js表示で、glbモデルが「逆立ちしている」ように見えたため、回転補正を追加しました。指摘されて削除し、また追加し、また削除する2往復をしました。実際はモデル自体がブレイクダンスの逆立ちフリーズポーズであり、元の向きが正しかったのです。

**教訓**: アセットの意味論（何を表しているか）は制作者にしか分からない。「異常」を感じて補正する前に「これは意図したものですか？」と1回確認する。

### ヒアリング文面にギフト券を勝手に追加した

ヒアリング依頼の文面を生成する際、ユーザーが一言も言及していない「Amazonギフト券1,000円分」を追加しました。

**教訓**: 外向けテキスト生成時は「ユーザーが言ったことだけで構成する」がデフォルト。コストが発生する要素は明示的な指示がない限り含めない。

### 「説明」を「実装依頼」と誤読した

ユーザーが「Page Annotationはウェブサイトから文章をハイライトして下部テキストボックスからノートを作成することでできると思います」と発言。これを実装依頼と解釈して機能を実装し、後でrevertしました。

**教訓**: 「〜できると思います」は説明であり指示ではない。実装前に「実装しますか？」と1行確認する。

## 分類3: 分析の軸偏り

### What軸偏り（2回再発）

企業調査のサブゴールをWhat軸（情報カテゴリ）だけで展開し、Who軸（ステークホルダー）・How軸（取引プロセス）・When軸（時間軸の変化）が構造的に漏れました。同じ失敗が5日で再発しました。

1回目はSansan企業調査、2回目はJSBシステムマップ生成。2回目は教訓記録とテンプレート更新後の再発でした。

**根本原因**: 1回目は「4軸展開ルール」がなかった。2回目は「リサーチのサブゴール設計」専用ルールだと思っていたが、システムマップのノード設計も本質的に同じ作業（変数リストを作る作業）だと気づいていなかった。

**対策**: 4軸展開ルールの適用範囲を「変数の設計全般」に拡大。「何かの変数リストを作るとき」が発火条件。

### データを根拠のない6分類で表示した

ダッシュボードのセグメントセクションで「受託デザイナー/イラスト/映像/ライター/フォト/VTuber」の6分類を使いました。分類基準は未定義でした。

**教訓**: データを分類するときは (1) 分類軸を先に定義 (2) 軸がMECEか確認 (3) 「この軸で切ると何の差が出るか」を1文で書く (4) データを当てはめる、の順。仮説がないなら大きさ順。

## 分類4: プロセス飛ばし

### Phase -1を飛ばしてテンプレートに機械照合した

テンプレート10項目への機械的照合だけでサブゴールマップを作り、Phase -1を実行しませんでした。テンプレートにない3観点が漏れていました。

**対策**: テンプレートの有無に関わらず、Phase -1は毎回実行する。テンプレートは出発点であり完成品ではない。

### ゴール分割を飛ばしていきなりタスク化した

「クリエイター業界をリサーチ」というゴールに対して、既存の4ストリームの上にいきなり乗ってレポート化しました。市場規模やバリューチェーンの角度が丸ごと抜けていました。

**対策**: タスク受領時は必ず「ゴール定義→サブゴールにMECE分割→タスク化→実行」の順を踏む。既存資産があっても先にゴール分割する。

## 分類5: ツール・情報源の過信

### WebFetchの出力を一次ソース確認として扱った

researcherが正しく報告していたブログ名を、WebFetchの出力に基づいて誤修正しました。WebFetchはHTMLのLLM要約であり、正確な文字列の確認には向きません。

**対策**: 固有名詞・タイトル・日付の確認は `curl -s | grep` でHTMLソースを直接検索するか、ブラウザ表示で目視。

### バックグラウンドagentの進捗を誤診断した

research-loopをバックグラウンドで起動し、中間ファイルが生成されていなかったため「停止」と誤診断して重複起動しました。実際はWebSearch主体で中間ファイルを作らない方式で正常稼働中でした。

**対策**: バックグラウンドagentが詰まったように見えても、まず完了通知を待つ。

## 分類6: 意思決定後の過剰介入

### 意思決定後に批判レビューを繰り返した

ユーザーがプランを選択した後に、追加の批判レビューサイクルを挟み、マイルストーン・撤退判定条件・達成定義の確認を返しました。「うるさい」と言われました。

**教訓**: 意思決定の前は批判的検証が必須。意思決定の後は実行支援に切り替える。追加の条件提示は不要。

### 「方向転換の回数」で延命を判定しようとした

「同じテーマで3週連続方向転換なら延命兆候」と書いたが、真摯に新証拠を取り込んだ正しい方向転換もカウントされてしまう。逆に1回目でも願望ベースなら延命。

**教訓**: 延命判定は「過去の結論を覆すに足る新証拠か、願望ベースの後付けか」で判断する。回数は指標にならない。

## 失敗ログの使い方

失敗ログは「読んで気をつける」ものではありません。各ログの「次にどうする」をCLAUDE.md・preferences.md・テンプレートに反映し、プロセスレベルで再発を防ぐためのインプットです。

失敗が発生 → lessons.mdに記録 → 「次にどうする」をルール化 → CLAUDE.mdまたはテンプレートに反映 → 次回以降のセッションで自動適用

このサイクルが回ることで、AI秘書は「壊れるたびに強くなる」設計になります。
