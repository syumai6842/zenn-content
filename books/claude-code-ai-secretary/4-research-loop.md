## 「何を調べるべきか」を調べる

リサーチで最も多い失敗は、いきなり検索を始めることです。「○○業界について調べて」と言われて即座にWebSearchを叩くと、検索上位の記事をつまみ食いしたレポートが出てきます。

本質的な問題は「何を調べるべきかを知らない状態で調べている」ことです。

この問題に対する解決策がメタリサーチ先行のプロトコルです。調べる前に「何を調べるべきか」を調べます。

## Phase -1: メタリサーチ（CEOが実行）

Phase -1はCEOの仕事です。目標を受け取ったら、まず以下を実行します。

### テンプレートの確認

`decomposition-templates.md` にゴールの種類に該当するテンプレートがあるか確認します。例えば「業界分析」なら以下のテンプレートがあります。

```
1. TAM/SAM/SOM 市場規模推計
2. 顧客セグメンテーション
3. 需要側pain/課題
4. 競合ポジショニングマップ
5. バリューチェーン/商流分析
6. Five Forces
7. PEST/制度環境
8. チャネルGTM
9. 海外ベンチマーク
10. トレンド/技術変曲点
```

しかしテンプレートはあくまで出発点です。毎回これで十分とは限りません。

### 動的補完

テンプレートに加えて、以下の検索を実行します。

```
「{リサーチの種類} フレームワーク」
「{research type} checklist」
「{research type} best practices」
「how to research {topic}」
```

見つかった記事をWebFetchで読み、テンプレートにない観点を抽出します。実際にこれを実行したところ、テンプレートにない3観点（スイッチングコスト、ボトムアップTAM、隣接プレイヤー拡張動向）が見つかったことがあります。

### 4軸展開

サブゴール候補を4つの軸からそれぞれ独立に生成します。

- **What軸**: 何を知りたいか（情報カテゴリ）
- **Who軸**: 誰が関わっているか（ステークホルダー）
- **How軸**: どう動いているか（オペレーション・プロセス）
- **When軸**: 時間軸でどう変化しているか

なぜ4軸か。1つの軸だけで展開すると、その軸では見えない項目が構造的に漏れるからです。What軸からはステークホルダーの行動実態が出てきません。Who軸からは市場規模の数字が出てきません。

この教訓は同じ失敗を2回（ある上場企業の調査と、別の企業のシステムマップ生成）繰り返してから確立されました。

## Phase 0: 情報源メタリサーチ（research-loopが実行）

Phase -1でサブゴールマップが確定したら、research-loopに委譲します。research-loopが最初にやるのはPhase 0、つまり「情報源の情報源を探す」ことです。

```
1. まとめ記事・リンク集を探す
2. 業界団体・公的機関のリソースページを探す
3. 上場企業リストを生成し、IRページURLを一覧化する
4. 学術・政府統計を探す
```

なぜIRページを明示的に含めるかというと、企業IRの決算説明会資料や有価証券報告書は通常の検索上位に来ませんが、TAM/SAM推計・セグメント別売上・競合認識が含まれている一次情報だからです。

Phase 0の成果物は「当たるべき情報源リスト」です。URLと「何が得られそうか」の1行メモをまとめたファイルを書き出します。

## Phase 1: researcher分配・並列起動

情報源リストができたら、サブゴールごとにresearcherを並列起動します。

researcherへの指示には5つの要素を含めます。

```yaml
questions: 回答すべき具体的な問い
sources: Phase 0で特定した情報源URLの割り当て
search_queries: 推奨する検索クエリ（最低8件）
scope: 他のresearcherとの分担境界
output: 出力先パス
```

「○○について調べて」では出力品質が安定しません。何を、どこから、どの範囲で調べるかを明示します。

独立したサブトピックのresearcherは1つのメッセージ内で並列起動します。3つのresearcherが同時に走れば、直列実行の1/3の時間で終わります。

## Phase 2: カバレッジ判定

全researcherの結果が返ってきたら、research-loopが各サブゴールのカバレッジを判定します。

```
covered:  事実・数値・出典が揃っている（2ソース以上）→ 1.0
partial:  一部の情報はあるが不足がある → 0.5
uncovered: まったく情報がない → 0.0
absent-unverified: 「存在しない」という結論だが深掘り未実施 → 0.5
```

**`absent-unverified`** は後述する失敗を踏まえて追加した状態です。「日本に○○は存在しない」という結論を、広範スキャンの段階でcoveredにしてしまうと、実際には存在するものを見逃します。

カバレッジスコア = (covered × 1.0 + partial × 0.5 + absent-unverified × 0.5) / サブゴール総数

## Phase 3: ループ判定

停止条件は3つ同時に満たす必要があります。

1. カバレッジスコア ≥ 80%
2. 前ラウンド比改善率 < 5%（情報飽和）
3. `absent-unverified` が0件

満たさなければPhase 1に戻ります。partial/uncoveredのサブゴールに対して追加のresearcherを起動し、`absent-unverified`のサブゴールに対しては業務フロー起点のクエリで深掘り検索を指示します。

小さなギャップ（1-2クエリで埋まる程度）はresearch-loop自身がWebSearchで補完します。

## Phase 4: 集約レポート

ループ終了後、全researcherの結果を1つのレポートに集約します。

集約時の原則は「**情報を落とさない**」です。要約で事実を削りません。削っていいのは重複（同一事実・同一出典の繰り返し）だけです。

集約レポートには以下を含めます。

```markdown
# 集約リサーチレポート

## メタ情報
- 実行ラウンド数
- 起動したresearcher数
- 最終カバレッジスコア
- 停止理由

## サブゴール別カバレッジ（表形式）

## 各サブゴールの収集結果（全事実・出典・数値）

## 矛盾するソース / 未解決の論点

## 出典一覧
```

このレポートがCEOに返ります。CEOは批判的検証を行い、問題があればresearch-loopを再起動します。

## 実例: 検証で覆った不在結論

海外トレンドの並列スキャンで「日本にある業種向けの特化SaaSは確認できない」とresearcherが報告しました。research-loopはこれをcoveredとして集約レポートに記載しました。

しかしCEOの検証で疑問が生じ、深掘りリサーチを指示したところ、業種特化の予約管理・顧客管理サービスが複数存在していると判明しました。

問題は「{業種名} SaaS 日本」のような抽象クエリでは業種特化の小規模SaaSは検索上位に来ないことでした。「{具体的な業務} 予約管理 システム」のような業務フロー起点のクエリが必要です。

この失敗が翌日にも再発しました（別の領域で「日本に特定サービスは不在」と判定したが、汎用プラットフォームの転用で同等機能が実現されていた）。2日連続で同じ構造の誤りが起きたことをきっかけに、`absent-unverified`状態とそのハンドリングルールが追加されました。

正しい問いは「日本に○○は存在するか」ではなく「**日本のユーザーは今この問題をどう解決しているか**」です。
